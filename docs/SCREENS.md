# Screens

実画面を作りながらデザインシステムを育てる方針。コンポーネント単体で作るより、実際の画面で当てはめる方がトークンやコンポーネントの抜けが浮き彫りになる（Linear / Vercel 系の社内デザインシステムもそう）。

## 候補一覧

| 画面 | これで試せること |
|---|---|
| **ログイン** | フォーム / バリデーション / ボタン / 小さいタイポ階層 |
| **設定画面** | タブ / フォームセクション / Switch / Combobox（Ark UI 系を一気に触れる） |
| **金融ダッシュボード** | テーブル / 数値カード / フィルタ / 密度の高い情報設計 |
| **メール（Gmail 風）** | サイドバー + リスト + 詳細の 3 ペインレイアウト / 検索 / ラベル |
| **カレンダー / スケジューラ** | DatePicker / Popover / グリッド（Ark UI の真価が出る複雑系） |
| **空状態 / 404 / エラー** | 「humble state」の作法。地味だけど design system でよく抜ける |
| **ランディングページ** | Hero、巨大タイポ、CTA（ダッシュボードとは別次元のフォントサイズ階層） |
| **チャット / Slack 風** | Avatar / 吹き出し / 時刻ラベル / プレゼンス感 |
| **価格表 / Pricing** | 比較カード、Toggle、Badge |
| **Kanban** | ドラッグ&ドロップ、列、カード |

## おすすめの順序

1. **ログイン** — small + 形が決まってる、トークンの初期化に最適。color / spacing / radius / font scale の土台がここで決まる
2. **設定画面** — Ark UI コンポーネントを一気に触れる
3. **金融ダッシュボード** — 密度設計、テーブル、数値
4. **メール** — レイアウト勝負、3 ペイン
5. **カレンダー** — 仕上げ、Popover / DatePicker
